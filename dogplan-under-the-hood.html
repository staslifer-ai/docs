DogPlan under the hood

A live dog-training platform at dogplan.net: curated training plans that adapt to a specific dog, a RAG-powered Q&A chat, a virtual pet with real game mechanics — built solo, end to end, with AI agents as the team.

What it is

DogPlan is not a chatbot that improvises training advice. Plans are curated content from a database, adapted to the dog by a rules engine — and AI is applied only where it is genuinely strong: semantic search and contextual answers.

Core decision · content over generation

Case Rules Engine: plans that adapt without an LLM

A training plan is a deep structure: plan → stages → exercises → text blocks. Generating it with an LLM would make quality unverifiable — a methodology risk you cannot ship to real dog owners.

How it works

Base plans live in the DB as templates. When a user describes their dog (fear, reactivity, age, puppy…), the engine clones the template and applies case rules — delete / replace / prepend / append operations targeted at a stage, an exercise or a single text block.

Why it matters

16 curated cases, 126 adaptation rules in production. Every adapted plan stays methodologically correct, because a cynologist curated both the base plan and the rules — the engine only transforms, it never invents.

AI where it belongs

RAG-powered Q&A — without LangChain

The in-plan chat answers user questions from a curated FAQ base. The whole RAG loop is raw SQL + asyncpg + pgvector — transparent, fast, debuggable.

The pipeline

Question → text-embedding-3-large → cosine search over pgvector (HNSW index) → top-3 FAQ above a 0.5 similarity threshold → gpt-4o composes the answer with the user’s plan as context.

The details

52 curated FAQ entries, embedded once at load time; Redis caches embeddings. No framework magic — every step is a SQL query or an explicit API call you can log and profile.

Game mechanics

Buddy: a virtual pet as a retention engine

Training a real dog earns XP for a virtual one. Buddy is a proper state machine — energy, mood, hunger and cleanliness degrade over time; levels, streaks, coins, minigames (fetch, tug-of-war, puzzle), 8+ trainable commands with mastery and cooldowns, and 3D avatars in Three.js.

Product engineering

The unglamorous parts that make it production

01

Passwordless auth

Email → 6-digit code (Redis, TTL) → JWT. No passwords for a mainstream audience of dog owners.

02

First-party analytics

Own tracker instead of Google Analytics: events batched (≤50), rate-limited, client IP stored only as a salted SHA-256 hash. Admin dashboard with DAU, sessions and per-session timelines.

03

Feature flags

Shop, game and knowledge base ship behind admin-managed flags — zero-downtime releases and reversible launches.

04

PWA, zero-build frontend

Vanilla JS SPA with a service worker, offline cache and install prompt. No bundler — the whole frontend deploys as static files.

05

i18n by design

Ukrainian + English across the stack: *_translations tables for every content type, locale JSON on the frontend, runtime placeholders like the dog’s name.

06

Async-first backend

FastAPI + asyncpg pool + Redis cache, GZip (60–70% payload cut), gunicorn/uvicorn behind nginx on a unix socket, systemd + Certbot in production.

Takeaway

What this case shows

One person can ship a full product — domain content model, RAG, game mechanics, analytics, auth, i18n, tests, deployment — when AI agents act as the team and the engineer keeps the architectural decisions: where AI belongs, and where curated content and deterministic rules do the job better.

Інженерний кейс · власний продукт

DogPlan під капотом

Жива платформа дресури собак на dogplan.net: кураторські плани тренувань, що адаптуються під конкретного собаку, RAG-чат запитань і відповідей, віртуальний улюбленець зі справжніми ігровими механіками — зібрано соло, від коду до продакшену, з AI-агентами в ролі команди.

Що це

DogPlan — не чат-бот, який імпровізує поради з дресури. Плани — це кураторський контент із бази даних, який адаптує rules-движок, а AI застосований лише там, де він справді сильний: семантичний пошук і контекстні відповіді.

Ключове рішення · контент, а не генерація

Case Rules Engine: плани, що адаптуються без LLM

План тренувань — глибока структура: план → етапи → вправи → текстові блоки. Генерувати його LLM-ом означало б неперевірювану якість — методологічний ризик, який не можна віддати реальним власникам собак.

Як працює

Базові плани лежать у БД як шаблони. Коли користувач описує свого собаку (страхи, реактивність, вік, цуценя…), движок клонує шаблон і застосовує case rules — операції delete / replace / prepend / append, націлені на етап, вправу або окремий текстовий блок.

Чому це важливо

16 кураторських кейсів, 126 правил адаптації у продакшені. Кожен адаптований план лишається методологічно коректним, бо і базовий план, і правила курував кінолог — движок лише трансформує, нічого не вигадує.

AI там, де йому місце

RAG-чат — без LangChain

Чат усередині плану відповідає на запитання з кураторської FAQ-бази. Весь RAG-цикл — це чистий SQL + asyncpg + pgvector: прозоро, швидко, зручно дебажити.

Пайплайн

Запитання → text-embedding-3-large → косинусний пошук у pgvector (HNSW-індекс) → top-3 FAQ з порогом схожості 0.5 → gpt-4o складає відповідь із планом користувача як контекстом.

Деталі

52 кураторські FAQ-записи, ембедяться один раз при завантаженні; Redis кешує ембединги. Жодної фреймворк-магії — кожен крок це SQL-запит або явний API-виклик, який можна залогувати і запрофілювати.

Ігрові механіки

Buddy: віртуальний улюбленець як механізм retention

Тренування реального собаки дає XP віртуальному. Buddy — повноцінна state-machine: енергія, настрій, голод і чистота деградують із часом; рівні, стріки, монети, мініігри (fetch, перетягування, пазл), 8+ команд із майстерністю та cooldown-ами і 3D-аватари на Three.js.

Продуктова інженерія

Негламурні частини, які роблять це продакшеном

01

Passwordless-автентифікація

Email → 6-значний код (Redis, TTL) → JWT. Без паролів — для масової аудиторії власників собак.

02

First-party аналітика

Власний трекер замість Google Analytics: події батчами (≤50), rate-limit, IP клієнта зберігається лише як salted SHA-256 хеш. Адмін-дашборд із DAU, сесіями і таймлайном подій.

03

Фіча-флаги

Магазин, гра і база знань виходять за адмін-керованими флагами — релізи без даунтайму і зворотні запуски.

04

PWA, zero-build фронтенд

Vanilla JS SPA із service worker, офлайн-кешем і встановленням як застосунок. Без бандлера — весь фронтенд деплоїться як статичні файли.

05

i18n від початку

Українська + англійська на всьому стеку: таблиці *_translations для кожного типу контенту, locale-JSON на фронтенді, run-time плейсхолдери на кшталт клички собаки.

06

Async-first бекенд

FastAPI + пул asyncpg + кеш Redis, GZip (−60–70% payload), gunicorn/uvicorn за nginx на unix-сокеті, systemd + Certbot у продакшені.

Висновок

Що показує цей кейс

Одна людина може відвантажити повний продукт — доменну модель контенту, RAG, ігрові механіки, аналітику, автентифікацію, i18n, тести й деплоймент — коли AI-агенти працюють як команда, а інженер лишає за собою архітектурні рішення: де місце AI, а де кураторський контент і детерміновані правила справляються краще.
