Needs-Driven Development

A methodology for building software with AI agents — where the unit of work is a user need, traced from a press release to shipped code, and enforced by an executable pipeline of specialized agents.

The idea in one line

Most teams keep vision, specification and delivery discipline in separate tools and separate heads. Needs-Driven Development fuses them into one traceable flow — and hands the whole flow to AI agents that are held to it by design.

Operationalized · the agent pipeline

The methodology, running as software

A graph state machine drives specialized AI agents through quality gates. Each stage passes named artifacts to the next; failures return to a specific node — not to the start.

PM→ PM·REV→ BA→ BA·REV→ DESIGN→ DEV→ DEV·REV→ QA→ QA·REV→ PERF→ COMMIT
Base agent — creates artifact
Review agent — checks it against injected skills
Commit gate — passes only if all gates clear
Auto-rollback — QA/PERF return to DEV

Double-pass review

Every artifact is created by one agent, then audited and fixed by a second agent carrying pluggable quality skills (OWASP, FastAPI, React, performance).

Contained rollback

Failures don't cascade

A QA fail or a critical perf finding sends the task back to DEV only — localizing the blast radius instead of re-running discovery.

Full audit trail

Every spec, wireframe, test result and diff is persisted and traceable back to the originating need.

Why it's defensible

What makes it distinctive

A

Requirements are executable

Acceptance criteria are written as Given/When/Then and reused verbatim as the QA test oracle — not aspirational prose.

B

Architecture maps to code

C4 diagrams carry a component-to-file table with NEW/EDIT markers, bridging design straight to implementation targets.

C

Value ships in vertical slices

Work is decomposed into end-to-end increments with local acceptance criteria — each independently testable.

D

Discovery is a structured pipeline

A four-stage funnel — interview, analysis, decomposition, confirmation — gated on a completeness checklist.

E

Quality rules are pluggable

Best-practice skills are external, per-project modules injected at review stages — not hard-coded rules.

F

The AI is kept on-need

Self-check and deviation control at every step guard against scope creep and drift from the original problem.

Операціоналізовано · конвеєр агентів

Методологія, що працює як ПЗ

Графова стейт-машина проводить спеціалізованих AI-агентів крізь гейти якості. Кожен етап передає іменовані артефакти далі; збої повертаються до конкретного вузла — не на початок.

PM→ PM·REV→ BA→ BA·REV→ DESIGN→ DEV→ DEV·REV→ QA→ QA·REV→ PERF→ COMMIT
Базовий агент — створює артефакт
Агент-рев'ю — перевіряє за інʼєктованими skills
Гейт коміту — проходить лише якщо всі гейти чисті
Авто-відкат — QA/PERF повертають до DEV

Подвійний прохід рев'ю

Кожен артефакт створює один агент, потім його аудитує і виправляє другий агент з підключними skills якості (OWASP, FastAPI, React, продуктивність).

Локалізований відкат

Збої не каскадять

Фейл QA чи критична знахідка perf повертають задачу лише до DEV — локалізуючи наслідки замість повторного discovery.

Повний аудит-слід

Кожна специфікація, wireframe, результат тесту й diff зберігаються і простежуються до початкової потреби.

Чому це захищувано

Що робить її особливою

A

Вимоги є виконуваними

Критерії приймання пишуться як Дано/Коли/Тоді й дослівно використовуються як тест-оракул QA — а не як побажання.

B

Архітектура мапиться на код

Діаграми C4 містять таблицю компонент→файл з маркерами NEW/EDIT, ведучи від дизайну прямо до цілей реалізації.

C

Цінність постачається зрізами

Роботу розбито на наскрізні інкременти з локальними критеріями приймання — кожен тестовний окремо.

D

Discovery — структурований конвеєр

Чотириетапна воронка — інтервʼю, аналіз, декомпозиція, підтвердження — з гейтом на чек-листі повноти.

E

Правила якості підключні

Skills з найкращими практиками — зовнішні модулі на рівні проєкту, інʼєктовані на етапах рев'ю, а не захардкоджені правила.

F

ІІ утримується на потребі

Самоперевірка й контроль відхилень на кожному кроці захищають від розповзання скоупу й дрейфу від початкової проблеми.
