The prequel to AI Workflow

From text instructions to a multi-agent platform

Before the AI Workflow platform there were plain text AI instructions — and they shipped a production admin panel that Comfy still runs. This is the story of the proof of concept, its limits, and why those limits demanded agents.

The claim

Platforms don’t appear from nowhere. AI Workflow started as a set of carefully written text instructions — effectively version zero of the platform, with no agents at all. It proved the core idea: AI can write production code, if you give it the right instructions and a real methodology.

The two methods

Spec first. User experience first.

The instruction set covered two complementary methodologies — one keeps the AI inside the architecture, the other keeps it inside the user’s problem.

01 · Spec Coding

Full specification first, code generation second. The instructions pin down the artifact structure, the API contract, naming rules and the code-generation format — so the AI stays inside the architecture instead of improvising one.

02 · Amazon Working Backwards

Start from the finished user experience — press release first, FAQ-driven design — and move backwards to the technical solution, with crisp problem definition and task decomposition on the way.

The proof

A production admin panel — with zero agents

What shipped

Through these text instructions alone, a production admin panel was built and is used by the company today. No multi-agent system, no orchestration — just instructions and methodology. That was the proof of concept.

What it proved

The bottleneck wasn’t the model’s ability to write code. It was everything around the code: holding context, reviewing quality, running parallel roles. Exactly the things a methodology encodes — and a single prompt can’t.

The limits

Where text instructions hit the wall

01

Context doesn’t hold

On a large project a single instruction set can’t keep the whole picture — the AI drifts from the architecture as the codebase grows.

02

Nobody reviews

The same “head” writes and checks the work. There is no independent reviewer for security, performance or business requirements.

03

No parallel roles

PM thinking, analysis, design, coding and testing are different jobs. One linear conversation forces them into one sequential blur.

04

Process lives in a person

The methodology worked because a human enforced it every time. It didn’t scale beyond its author — the process had to become the system.

The evolution

Instructions became agents

Every limit mapped to a design decision in the next system: context → named artifacts passed between stages; no reviewer → a dedicated review agent at every stage; no parallel roles → specialized agents with clear roles; process-in-a-person → a pipeline that enforces the methodology itself. That system is AI Workflow.

Приквел до AI Workflow

Від текстових інструкцій до multi-agent платформи

До платформи AI Workflow були звичайні текстові AI-інструкції — і вони відвантажили production-адмінку, якою Comfy користується досі. Це історія доказу концепції, його меж — і чому ці межі вимагали агентів.

Теза

Платформи не з’являються з нізвідки. AI Workflow починався як набір ретельно написаних текстових інструкцій — фактично нульова версія платформи, взагалі без агентів. Вони довели головне: AI може писати production-код, якщо дати йому правильні інструкції та справжню методологію.

Два методи

Спочатку специфікація. Спочатку користувач.

Набір інструкцій покривав дві взаємодоповнювальні методології: одна тримає AI у рамках архітектури, друга — у рамках проблеми користувача.

01 · Spec Coding

Спочатку повна специфікація, потім генерація коду. Інструкції фіксують структуру артефактів, контракт API, правила іменування та формат генерації коду — тож AI лишається всередині архітектури, а не імпровізує власну.

02 · Amazon Working Backwards

Почати з готового користувацького досвіду — press release first, FAQ-driven design — і рухатися назад до технічного рішення, з чітким визначенням проблеми та декомпозицією задач дорогою.

Доказ

Production-адмінка — без жодного агента

Що відвантажили

Лише через ці текстові інструкції була збудована production-адмінка, якою компанія користується сьогодні. Без multi-agent системи, без оркестрації — тільки інструкції та методологія. Це й був доказ концепції.

Що це довело

Вузьким місцем була не здатність моделі писати код. А все навколо коду: утримання контексту, рев’ю якості, паралельні ролі. Саме те, що кодує методологія — і чого не вміє один промпт.

Межі

Де текстові інструкції вперлися в стіну

01

Контекст не тримається

На великому проєкті один набір інструкцій не втримує всю картину — AI дрейфує від архітектури разом зі зростанням кодової бази.

02

Ніхто не рев’юїть

Та сама «голова» і пише, і перевіряє. Немає незалежного рев’ювера на security, performance чи бізнес-вимоги.

03

Немає паралельних ролей

PM-мислення, аналіз, дизайн, код і тестування — різні роботи. Одна лінійна розмова змішує їх в один послідовний потік.

04

Процес живе в людині

Методологія працювала, бо людина щоразу її дотримувалась. Це не масштабується поза автором — процес мав стати системою.

Еволюція

Інструкції стали агентами

Кожна межа перетворилася на дизайн-рішення наступної системи: контекст → іменовані артефакти між етапами; відсутність рев’ювера → окремий review-агент на кожному етапі; відсутність паралельних ролей → спеціалізовані агенти з чіткими ролями; процес-у-людині → пайплайн, який сам виконує методологію. Ця система — AI Workflow.
